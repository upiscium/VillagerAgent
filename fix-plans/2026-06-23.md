CRAFT Dual-DAG Clarification Policy 分析・改善指示書

1. 対象

Repository: https://github.com/upiscium/VillagerAgent.git
Base branch: main

作業開始時に最新の "main" を取得し，既存の CRAFT Dual-DAG 実装，設定，テスト，分析コード，既存 findings を確認すること．

本指示書の設定名やファイル名が現在の実装と異なる場合は，既存設計との整合性を優先すること．既存機能を重複実装しないこと．

2. 背景

現在，CRAFT における VillagerAgent + Dual-DAG 条件の progress が baseline より低い．

途中分析から，主要因として "Clarify" が物理行動ターンを消費し，action throughput を低下させている可能性が高い．

現状の Clarify policy は主に以下を根拠としている．

- action confidence
- conflict count
- required evidence
- large block span uncertainty
- clarification cost
- mistake cost

しかし，CRAFT の oracle-assisted 条件では，Builder に提示される oracle candidate は現在実行可能で forward progress を生む候補である．

そのため，単純な uncertainty-based gating により Clarify を選ぶと，

進捗可能な oracle action を見送る
→ Clarify が1ターン消費する
→ action throughput が低下する
→ final progress が低下する

という構造が発生する．

本タスクでは Clarify を単純に削除するのではなく，Clarify がいつ有益で，いつ不要または有害かを計測し，uncertainty-based policy から value-of-information-based policy へ改善する．

3. 研究上の前提

公式 CRAFT 条件は次である．

craft:
  use_oracle: true
  oracle_n: 5

run:
  turns: 20

この条件を主実験として維持すること．

次の条件は補助実験としてのみ使用する．

oracle_n = 1:
  Clarify がほぼ支配される条件の診断

oracle_n = 3:
  oracle candidate 数の中間的な感度分析

turns = 30:
  Clarify の情報価値を長い horizon で回収できるかの分析

"oracle_n=3", "turns=30" を公式条件の代替として扱ってはならない．

4. 重要な仮説

次の仮説を明示的に検証する．

H1: Action throughput hypothesis

Dual-DAG は物理行動1回当たりの質を悪化させていないが，Clarify により物理行動回数が減っている．

判定例:

Dual-DAG mean_progress_delta_per_physical_action
  >= baseline

Dual-DAG mean_progress_delta_per_turn
  < baseline

Dual-DAG physical_action_count
  < baseline

H2: Oracle dominance hypothesis

有効な oracle candidate が存在する場合，Clarify の期待価値は物理行動の機会費用を下回る．

H3: Same-action hypothesis

Clarify 前後で，最終的に選ばれる物理行動が同一であるケースが多い．

この場合，Clarify は行動を改善せず，単に1ターン遅らせている．

H4: Failed-resolution hypothesis

Clarification response が得られても，Hypothesis，ResolvedFact，required evidence，candidate state が十分に変化していない．

H5: Short-horizon amplification hypothesis

5ターン条件では Clarify の固定コストが過大に評価され，20ターンまたは30ターンでは相対的な悪影響が小さくなる．

H6: Oracle-candidate-count hypothesis

"oracle_n=1" では Clarify はほぼ不要だが，"oracle_n=3" または "oracle_n=5" では candidate discrimination に利用できる可能性がある．

ただし，oracle candidate は複数存在しても全て forward progress 候補であるため，progress 上の Clarify の利益は限定的である可能性を考慮すること．

6. Phase B: Clarify outcome classification

各 Clarify event を offline analysis で次のいずれかに分類する．

Beneficial

最低限，次のいずれかを満たす．

- blocked candidate が executable になった
- 誤った上位 candidate が invalidated された
- top candidate が変化し，その後の物理行動が正の progress を生んだ
- Clarify が複数の将来 action candidate をアンロックした
- action parse または実行不能状態を解消した

Neutral

- graph state は変化したが top action は変化しなかった
- 結局 Clarify 前と同じ action を実行した
- confidence だけが変化し，実行判断に影響しなかった

Harmful

- Clarify 前に実行可能な oracle candidate があった
- Clarify 後も同じ candidate を実行した
- Clarify によって残り物理行動数が減った
- Clarify 後に progress が改善しなかった

Failed

- response が得られなかった
- response parse に失敗した
- ReportedClaim / ResolvedFact が生成されなかった
- required evidence が解決されなかった
- candidate state が変化しなかった
- 同じ質問が再度生成された

分類ルールは決定的かつテスト可能にすること．

主観的な LLM 判定を必須にしないこと．

8. Phase D: Clarification opportunity cost

Clarify の機会費用を offline analysis で推定する．

Runtime agent に hidden target を渡してはならない．

推奨する機会費用の推定方法は次の優先順位とする．

方法1: Oracle candidate progress

Clarify 前に有効な oracle candidate が存在する場合，その candidate の simulation または oracle metadata に含まれる progress を利用する．

turn_opportunity_cost =
    Clarify 前の最上位 valid oracle candidate の expected progress delta

方法2: Matched baseline

同じ structure，seed，turn，board state に近い baseline trajectory から物理行動の平均 progress delta を推定する．

方法3: Empirical action value

同一条件の過去の正の物理行動から平均 delta を推定する．

使用した推定方法を artifact に記録すること．

Runtime decision に evaluator-only hidden informationを使用してはならない．

10. Value of Clarification の定義

基本形は次とする．

expected_clarification_value =
    expected_avoided_mistake_cost
  + expected_future_unlock_value
  - turn_opportunity_cost
  - clarification_execution_cost
  - clarification_failure_risk

簡易実装例:

p_wrong =
    uncertainty estimate based on confidence and candidate margin

p_resolution =
    estimated probability that the question resolves the missing evidence

expected_avoided_mistake_cost =
    p_wrong
    * p_resolution
    * mistake_cost

expected_future_unlock_value =
    p_resolution
    * expected_unlocked_candidate_count
    * mean_candidate_value

value_of_clarification =
    expected_avoided_mistake_cost
    + expected_future_unlock_value
    - turn_opportunity_cost
    - clarification_cost

"confidence" は校正された確率とは限らないため，結果を確率として過剰解釈しないこと．

各構成要素をログに残し，後から重みを分析できるようにすること．

11. Oracle-aware hard rules

Value-of-information の前に，次の hard rule を適用する．

Rule 1: Oracle candidate が1つ

if oracle_enabled
and valid_oracle_candidate_count == 1
and candidate is syntactically executable:
    Clarify を禁止
    candidate を実行

例外:

- action parse に必要な引数が欠損
- coordinate が不正
- layer が不正
- large block span が action representation 上欠損
- candidate を runtime action に変換できない

単なる低 confidence，claim conflict，複数 Director の合意不足を例外にしないこと．

Rule 2: 複数 oracle candidate

複数候補がある場合でも，次を全て満たす場合だけ Clarify を許可する．

top1-top2 margin が閾値未満
質問対象となる evidence が明確
質問によって top candidate ranking が変化し得る
対象 Director が特定できる
clarification budget が残っている
十分な残りターンがある
value_of_clarification > 0

Rule 3: Oracle candidate は全て forward progress

複数候補が全て進捗候補であることを考慮し，単に"どれが正しいか分からない"だけで Clarify しないこと．

Clarify は次の場合に限定する．

- candidate の実行可能性が異なる
- Director の記述との整合性を研究指標として改善する明確な価値がある
- 質問によって複数の後続 candidate がアンロックされる
- 重大な lifecycle inconsistency がある

progress 最適化モードでは，valid oracle candidate の実行を原則優先すること．

次の config を追加する．

dual_dag:

    prevent_duplicate_clarifications: true

推奨 default は，既存挙動との後方互換性を考慮して決めること．

実験 config では明示的に値を指定すること．

13. Duplicate clarification suppression

canonical question key を定義する．

推奨要素:

target_director
coordinate
layer
block_attribute
span
question_type
target_candidate_ids
required_evidence_ids

同じ canonical key に対して，以下が変化していない場合は再質問を禁止する．

- relevant graph state
- board revision
- candidate set
- required evidence
- target Director response

質問文の表面的な言い換えだけで duplicate 判定を回避できないようにすること．

14. Late clarification suppression

Clarify 後の回答を利用する物理行動ターンが残らない場合は Clarify を禁止する．

最低限次を判定する．

remaining_turns
expected_response_latency
minimum_followup_action_count

例:

remaining_turns <= expected_response_latency:
    Clarify 禁止

5ターン smoke では，終盤 Clarify がほぼ必ず不利になることを考慮する．

15. Expected unlock requirement

Clarify candidate は次を明示的に参照しなければならない．

target_candidate_ids
required_evidence_ids
expected_candidate_state_transitions
target_director

次のような一般的質問は生成しない．

"もう少し詳しく説明してください"
"位置を確認してください"

質問は具体的でなければならない．

例:

D2に対して，
global coordinate (1,2), layer 1 の blue large block が
(1,1) まで span するかを確認する．
回答により candidate A を executable，
candidate B を invalidated にする予定．

16. WaitForEvidence の制限

CRAFT では，何も行動しなければ新しい evidence が自然発生しない場合がある．

次のどちらも存在しない場合，"WaitForEvidence" を禁止する．

pending clarification response
scheduled observation or response event

"WaitForEvidence" が単なる保守的 no-op にならないようにすること．

22. テスト要件

Clarify tracing

- Clarify 前後の candidate snapshot が保存される
- 次の physical action と関連付けられる
- same-action 判定が正しい

Outcome classification

- beneficial
- neutral
- harmful
- failed

それぞれの代表ケースをテストする．

Oracle-aware policy

- valid oracle candidate が1つなら Clarify しない
- candidate が実行不能なら Clarify を許可できる
- oracle candidate が複数でも margin が大きければ Clarify しない
- margin が小さくても質問が ranking に影響しなければ Clarify しない
- non-oracle mode の挙動を壊さない

Budget

- episode budget を超えない
- cooldown が機能する
- late clarification が抑制される
- duplicate clarification が抑制される

WaitForEvidence

- pending source がなければ選択不可
- pending response があれば選択可能

Metrics

- progress@20 / progress@30
- normalized AUC
- clarification efficiency
- opportunity cost
- candidate count sensitivity

Safety

- hidden target を runtime prompt に渡さない
- oracle internal metadata を Director に渡さない
- private view を他 Director に渡さない
- evaluator-only analysis を public Dual-DAG artifact に混入させない

25. 禁止事項

- "oracle_n=3, turns=30" のみを主結果として採用しない
- 公式条件を都合よく変更して baseline を上回ったと主張しない
- 5ターン smoke だけで結論を出さない
- Clarify を無条件に削除して終わらせない
- confidence threshold を根拠なく調整しない
- oracle candidate 数を誤答候補数として扱わない
- hidden target を runtime policy に使用しない
- evaluator-only情報を public artifact に漏らさない
- progress 改善だけを報告し，communication alignment の悪化を隠さない
- 実験できなかった結果を推測で書かない
