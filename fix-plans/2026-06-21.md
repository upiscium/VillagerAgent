# CRAFT Dual-DAG Progress Regression の原因分析・改善指示書

## 1. 対象リポジトリ

```text
https://github.com/upiscium/VillagerAgent.git
```

作業開始時点の `main` を基準とすること．

最初に現在のコード，設定，テスト，既存レポートを確認し，この指示書に記載されたファイル名や実装状況が現在の `main` と異なる場合は，リポジトリの現状を優先すること．

## 2. 背景

CRAFT 向けに Dual-DAG アーキテクチャを実装している．

主要要素は以下である．

* Epistemic DAG
* Action Candidate DAG
* `ObservedFact`
* `PublicFact`
* `ReportedClaim`
* `Hypothesis`
* `ResolvedFact`
* support / conflict / required evidence
* graph traversal による action unlock
* gated clarification
* historical public evidence retrieval
* clarification response ingestion
* action candidate lifecycle
* `Clarify` / `WaitForEvidence`
* CRAFT action candidate から VillagerAgent Task DAG への projection

現在，Dual-DAG 条件の progress が baseline 条件より低い結果が観測されている．

本タスクの目的は，単に閾値を調整して一時的にスコアを上げることではない．比較条件を公正にした上で，progress 低下が次のどこに起因するかを切り分け，根拠に基づいた最小限の修正を実装することである．

1. 実験設定の不一致
2. Clarify / Wait による行動スループット低下
3. oracle candidate に対する過剰な gating
4. historical retrieval による stale evidence
5. evidence summary による prompt interference
6. single-source evidence を解決できない過剰 consensus
7. false conflict
8. confidence の非校正
9. Builder fallback
10. lifecycle 更新または run-loop 統合の不具合

## 3. 最重要の既知問題

### 3.1 Oracle candidate 数が一致していない

現在の設定には，少なくとも以下の差がある．

```text
Official baseline:
  oracle_n: 5

VillagerAgent baseline:
  oracle_n: 1

VillagerAgent + Dual-DAG:
  oracle_n: 1
```

Official baseline と Dual-DAG の差には，Dual-DAG の有無だけでなく，VillagerAgent director pipeline の有無と oracle candidate 数の差が含まれている．

この条件を"Dual-DAG 対 baseline"の因果比較として扱ってはならない．

### 3.2 評価ターン数が短い

現在の `gemma4_12b_progress_large` は名前に反して，主に次の条件である．

```text
structures: 0..4
seeds: 1, 3, 5
turns: 5
```

5ターン評価は smoke test や早期挙動の診断には利用できるが，CRAFT の最終 progress 比較には不十分である．

Clarify，初期 graph 構築，初期 hypothesis 生成などの固定オーバーヘッドが，短い horizon では過大に評価される．

### 3.3 実装状況文書が古い可能性がある

Issue #92～#96 相当の機能は現在の main に既に統合されている可能性が高い．

作業開始時に以下を確認すること．

* artifact schema versioning
* CRAFT run loop への unlock / hypothesis lifecycle 接続
* Minecraft runtime task selection
* lifecycle metrics
* Builder action 後の `executed` 遷移

実装済みであれば，未実装として再実装しないこと．関連する proposal / roadmap / implementation status 文書を現状に合わせて更新すること．

## 4. 基本方針

以下の順序を厳守すること．

```text
比較条件の正常化
→ 計測追加
→ アブレーション
→ 原因判定
→ 修正
→ 再評価
```

原因計測前に，推測だけで gate threshold や scoring weight を変更してはならない．

CRAFT の既存 `overall_progress` の定義は，このタスクでは変更しないこと．比較可能性を維持し，必要な診断指標を別途追加すること．

## 5. Task A: 比較条件の監査と正常化

### 5.1 比較条件を3系統に分ける

実験条件を明確に分離すること．

#### Condition O: Official CRAFT baseline

```text
villageragent.enabled: false
dual_dag.enabled: false
oracle_n: 5
```

これは公式実装に近い参照条件である．

#### Condition V: VillagerAgent baseline

```text
villageragent.enabled: true
dual_dag.enabled: false
```

#### Condition D: VillagerAgent + Dual-DAG

```text
villageragent.enabled: true
dual_dag.enabled: true
```

Dual-DAG の因果効果を評価する主要比較は，必ず次とする．

```text
Condition V vs Condition D
```

Condition O と Condition D の差を，Dual-DAG 単体の効果として解釈してはならない．

### 5.2 Oracle parity

Condition V と Condition D では，以下を完全に一致させること．

* `oracle_n`
* `use_oracle`
* structures
* turns
* seed
* Director model
* Builder model
* model provider
* temperature
* max tokens
* tool use
* common ground 設定
* partial observation 設定
* VillagerAgent component flags
* prompt history policy
* Director ordering policy

Official baseline とも比較したい場合は，別途次の2系列を作ること．

```text
oracle_n = 1 の O / V / D
oracle_n = 5 の O / V / D
```

ただし，主たる Dual-DAG アブレーションでは V と D を使用する．

### 5.3 Paired config validator

比較対象設定間の意味的な差を検査する validator を追加すること．

例:

```text
python -m benchmarks.craft.validate_comparison_configs \
  --baseline <V config> \
  --treatment <D config>
```

Validator は次を出力する．

* 一致しているフィールド
* 異なるフィールド
* 許可された差
* 許可されていない差
* validation pass / fail

V と D の間で原則許可する差は以下のみとする．

```text
run.name
run.output_dir または suffix
dual_dag.enabled
dual_dag 配下の設定
```

Dual-DAG 実装上どうしても必要な差がある場合は allowlist に明示し，理由をコードコメントと文書に残すこと．

### 5.4 ターン数

実験を次の2段階に分ける．

#### Diagnostic smoke

```text

seeds: [1, 3, 5]
turns: 5
```

目的:

* artifact 生成確認
* runtime error 確認
* gate reason 確認
* fallback 確認
* 短時間アブレーション

#### Full evaluation

最低限，次を用意する．

```text

seeds: 5個以上
turns: 20
```

コスト上すぐに全実行できない場合でも，再現可能な manifest，実行コマンド，期待出力パスを作成すること．

5ターンの結果を最終性能として報告しないこと．

## 8. Task D: 原因判定ロジック

アブレーション結果から，少なくとも次の判定を行うレポートを生成すること．

### 8.1 行動スループット低下

次を満たす場合:

```text
D mean_progress_delta_per_physical_action >= V
D mean_progress_delta_per_turn < V
D physical_action_count < V
```

判定:

```text
主要原因は action quality ではなく action throughput の低下
```

### 8.2 Clarification overhead

次を満たす場合:

```text
clarify_count > 0
clarification_to_unlock_rate が低い
または
clarification_to_positive_action_latency が大きい
```

判定:

```text
Clarify が progress に変換されていない
```

### 8.3 Prompt interference

次を満たす場合:

```text
C2 < C0
かつ
C1 ≒ C0
```

判定:

```text
graph bookkeeping ではなく evidence summary が Builder を悪化させている
```

### 8.4 Historical retrieval pollution

次を満たす場合:

```text
C3 < C2
```

追加で確認:

* retrieved node age
* executed candidate retrieval
* superseded claim retrieval
* top action が retrieval により変更された割合

判定:

```text
historical retrieval に stale evidence または context pollution がある
```

### 8.5 Gating overhead

次を満たす場合:

```text
C4 < C3
```

判定:

```text
gating 自体が正しい候補を抑制している
```

### 8.6 Coordination action overhead

次を満たす場合:

```text
C5 < C4
```

判定:

```text
Clarify によるターン消費が主要因
```

### 8.7 Official baseline confound

Official baseline が V/D より高くても，`oracle_n`，VillagerAgent pipeline，prompt，Director ordering のいずれかが異なる場合は，Dual-DAG 原因と結論づけないこと．

## 9. Task E: Oracle-aware gating の実装

計測とアブレーションにより gating が主要原因と確認された場合に実装すること．

独立した config flag を追加する．

例:

```yaml
dual_dag:
  gated_clarification:
    oracle_aware: true
```

### 9.1 基本原則

oracle verified candidate は，CRAFT evaluator により forward progress 候補として提供されている．

したがって oracle 条件では，次だけを理由に candidate を block してはならない．

* absolute confidence が閾値未満
* 複数 Director の corroboration がない
* required evidence metadata が残っている
* historical claim conflict がある

ただし，candidate の action syntax が不完全，実行引数が欠損，候補を一意に選べないなど，実際に action を実行できない場合は例外とする．

### 9.2 推奨ルール

```text
if oracle_verified candidates exist:
    remove candidates that are syntactically invalid
    remove candidates that conflict with current public board state
    rank remaining candidates using evidence score

    if top candidate exists:
        execute top candidate by default

        clarify only if:
            top1-top2 margin < configured margin threshold
            and clarification target is known
            and clarification is expected to distinguish top1/top2
            and clarification budget remains
```

Absolute confidence だけではなく，候補間 margin を使用できるようにすること．

### 9.3 Non-oracle 条件

Non-oracle 条件では，従来の mistake cost と required evidence を用いた保守的 gate を維持できるようにする．

Oracle-aware behavior を non-oracle 条件へ暗黙に適用しないこと．

## 10. Task F: Clarification policy の安全弁

計測結果に関係なく，clarification loop の防止機構を config-gated に追加すること．

推奨デフォルト:

```yaml
dual_dag:
  gated_clarification:
    max_clarifications_per_episode: 2
    clarification_cooldown_turns: 2
    prevent_duplicate_clarifications: true
```

### 10.1 Duplicate suppression

同じ canonical clarification key に対して，回答または graph state の実質的変化がないまま再質問してはならない．

### 10.2 Clarification target

Clarify を生成する場合は，最低限次を明示すること．

* target Director
* unresolved coordinate
* unresolved layer
* unresolved block attribute または span
* 何が分かればどの candidate が unlock されるか

一般的な"もう少し詳しく説明してください"のような質問は禁止する．

### 10.3 WaitForEvidence

CRAFT では外部 evidence が自然発生しない可能性が高い．

そのため `WaitForEvidence` は，最低限次のどちらかを満たす場合だけ選択可能とする．

```text
pending clarification response が存在する
明示的に予定された observation / response event が存在する
```

新しい evidence の発生元が存在しない場合，`WaitForEvidence` を選んではならない．

## 11. Task G: Single-source evidence の扱い

CRAFT では，特定のセルについて有効な情報を持つ Director が1人だけである場合がある．

そのため，単純な"複数 Director の claim 一致"を unlock の必須条件にしてはならない．

Director の直接観測を次のように評価できるようにすること．

```text
evidence_strength =
    source_reliability
    × visibility_relevance
    × spatial_specificity
    × temporal_freshness
```

少なくとも次の場合は，single-source evidence で candidate を unlock 可能にする．

```text
claim source が当該 coordinate を観測できる Director
claim が direct observation 由来
claim が十分に具体的
current board state と明示的に矛盾しない
他の観測可能 Director から明示的 conflict がない
```

他 Director の沈黙または非観測を conflict として扱わないこと．

この変更も config-gated にし，アブレーション可能にすること．

## 12. Task H: Stale evidence 防止

各 relevant node に既存フィールドを再利用するか，必要に応じて次を追加する．

```text
valid_from_turn
valid_until_turn
board_revision
consumed_by_action
superseded_by
```

Historical retrieval ではデフォルトで次を除外または強く減衰する．

* executed action candidate
* invalidated candidate
* superseded claim
* 現在の board revision と整合しない claim
* clarification により解決済みの古い uncertainty
* 同じ action により consumed 済みの命令的 claim

既存 schema versioning 方針に従い，artifact schema を破壊的に変更しないこと．

必要なら後方互換な optional field として追加する．

## 13. Task I: Offline diagnostic evaluator

ただし，実験終了後の offline analysis では，原因診断のために evaluator-only data を使用してよい．

Offline analysis で可能なら次を計測する．

```text

blocked_candidate_was_forward_progress
conflict_edge_was_world_state_conflict
```

* agent runtime から参照不可能にする
* prompt artifact と分離する
* partial-information leakage validator の対象を維持する
* public Dual-DAG artifact に hidden target を書き込まない

## 16. テスト要件

最低限，次を追加または更新する．

### Config parity

* V/D の許可された差だけなら pass
* `oracle_n` が異なれば fail
* turns が異なれば fail
* model が異なれば fail
* temperature が異なれば fail

### Metrics

* physical action count
* Clarify count
* Wait count
* progress AUC
* delta per turn
* delta per physical action
* gate reason aggregation
* candidate lifecycle aggregation

### Clarification

* duplicate clarification suppression
* clarification budget
* cooldown
* target Director がない質問の拒否または fallback
* graph state が変わった後は必要に応じて再質問可能

### WaitForEvidence

* pending source なしでは選択不可
* pending clarification がある場合は選択可能

### Oracle-aware gate

* oracle verified candidate が低 confidence だけで block されない
* syntactically invalid candidate は除外される
* margin が十分なら Clarify しない
* margin が小さくても budget なしなら action を選ぶ
* non-oracle mode の既存挙動を壊さない

### Retrieval

* executed candidate を再取得しない
* invalidated candidate を再取得しない
* superseded claim を除外する
* current-turn evidence が優先される

### Safety

* hidden target が prompt に入らない
* oracle moves が Director に入らない
* private view が他 Director に漏れない
* evaluator-only diagnostics が runtime artifact に混入しない

## 19. 実装上の禁止事項

* 原因分析前に閾値だけを恣意的に変更しない
* Official baseline と Dual-DAG の差をそのまま Dual-DAG 効果と呼ばない
* 5ターンだけで最終性能を結論づけない
* progress metric の定義を都合よく変更しない
* runtime agent に hidden target を渡さない
* offline evaluator 情報を public DAG に混入させない
* 既存の schema versioning を無視しない
* 実装済み機能を重複実装しない
* 実験未実行の結果を推測で記載しない
* テストを削除または弱体化して通さない

## 20. 作業の優先順位

```text
P0:
  現状確認
  proposal の実装状況確認
  oracle_n / turns / model / seed の parity 修正
  config parity validator

P1:
  throughput / gate / clarification / retrieval metrics
  ablation configs
  5-turn diagnostic smoke

P2:
  原因判定
  oracle-aware gate
  clarification budget / duplicate suppression
  WaitForEvidence restriction

P3:
  single-source unlock
  stale evidence filtering
  full 20-turn evaluation
  findings と proposal 更新
```

最初の実装では，P0 と P1 を優先し，原因が確認できないまま P2/P3 の挙動変更を大量に追加しないこと．
